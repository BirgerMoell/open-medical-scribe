This document is the starter clinical evaluation plan for the OMS Class I Web Reference Build.

It is written as a planning document for an ordinary Class I medical device software pathway. It is not the clinical evaluation report.

## Purpose

To define how the manufacturer will establish and maintain sufficient clinical evidence that the regulated web build is safe and performs as intended for its stated intended purpose.

## Device under evaluation

- OMS Class I Web Reference Build
- manufacturer-controlled web SaaS
- intended purpose limited to transcript-grounded clinical documentation support for professional users

## Clinical evaluation objective

Demonstrate that:

1. the product performs consistently with its intended documentation-support purpose
2. the review-first workflow and product constraints reduce the risk of unsafe use
3. residual risks are acceptable when weighed against the intended documentation-support benefit

## Intended users and environment

- licensed healthcare professionals
- authorized support staff under clinician supervision where permitted
- professional healthcare settings and controlled remote documentation workflows

## Clinical claims to support

Claims that require evidence:

- encounter audio can be converted into transcript-grounded draft documentation
- the software supports clinician documentation workflows
- the clinician remains the final reviewing and approving user

Claims not to support:

- improved diagnostic accuracy
- improved treatment decisions
- triage support
- referral support
- prognostic or predictive claims

## Planned evidence sources

### 1. Device-specific validation evidence

- intended-use validation of the regulated workflow
- hazard-based validation scenarios
- usability validation focusing on review and approval behavior
- content-grounding assessment against representative transcripts

### 2. Literature and state-of-the-art review

- literature on clinical documentation support workflows
- literature on human review of AI-generated documentation
- relevant state-of-the-art software and safety guidance

### 3. Post-market and complaint data

- user complaints
- support tickets
- PMS findings
- incident assessments

## Key clinical questions

1. Does the product support documentation tasks as intended for representative users?
2. Can intended users reliably distinguish draft from approved output?
3. Are key residual risks such as hallucinated or omitted content sufficiently controlled by the workflow and product design?
4. Does the product remain within its non-decision-support intended-purpose boundary in practice?

## Evaluation approach

The initial evaluation should combine:

- review of the intended purpose and risk profile
- review of validation evidence from representative use scenarios
- literature review sufficient for the documentation-support context
- benefit-risk analysis tied to the risk file

## PMCF planning note

For this Class I software pathway, PMCF should still be considered. The initial plan may be limited and proportional, but it should define:

- what post-market clinical information will be gathered
- how complaint and use data will be reviewed
- what triggers a clinical-evidence update

## Outputs

1. clinical evaluation report
2. PMCF plan or justified PMCF rationale
3. updates to risk management and PMS where needed

## Immediate next tasks

1. Define the literature search strategy.
2. Define the intended-use validation study design.
3. Define the transcript-grounding review rubric.
4. Link clinical questions to the hazard analysis and V&V plan.
