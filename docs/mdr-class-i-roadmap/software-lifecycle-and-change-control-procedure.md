# Software Lifecycle And Change Control Procedure

Version: 0.1
Status: Draft
Owner: Birger Moëll

Last updated: 2026-03-11

This procedure defines how the OMS Class I Web Reference Build is designed, released, changed, and maintained under manufacturer control.

## Purpose

To ensure that the regulated build remains consistent with its validated intended purpose and that changes are assessed before release.

## Scope

Applies to:

- application code
- infrastructure configuration
- browser-client workflow behavior
- prompts
- models
- providers
- release packaging
- production runtime settings

## Lifecycle states

1. Requirement defined
2. Design updated
3. Risk reviewed
4. Implementation completed
5. Verification completed
6. Validation completed where required
7. Release approved
8. Production monitored

## Controlled baselines

The regulated product must maintain controlled baselines for:

- source code release
- production configuration
- supported environment definition
- prompt set
- model/provider selection
- IFU and labeling

## Change categories

### Category A: administrative

Examples:

- typo fixes in internal documents
- non-regulated documentation cleanup

Expected handling:

- record the change
- no product verification impact unless regulated labeling is affected

### Category B: minor product change

Examples:

- UI clarification that does not alter intended use or workflow logic
- non-safety-critical bug fixes

Expected handling:

- impact assessment
- targeted verification
- update release record

### Category C: significant regulated change

Examples:

- workflow changes affecting review or approval
- prompt changes affecting note output behavior
- model version change
- provider change
- authentication or audit behavior change
- browser support expansion

Expected handling:

- formal impact assessment
- risk review
- verification and, where needed, validation update
- documentation update before release approval

### Category D: classification-critical change

Examples:

- adding diagnostic, triage, treatment, referral, or urgency-related output
- enabling self-hosted deployment inside the CE-marked claim
- enabling end-user model or provider switching
- expanding to a materially different platform configuration

Expected handling:

- stop ordinary change process
- re-run qualification and classification analysis
- update intended purpose if needed
- determine whether a new conformity assessment path is required

## Mandatory change assessment questions

For every non-administrative change, assess:

1. Does the change affect intended purpose, users, or use environment?
2. Does the change affect transcript grounding or draft-review workflow?
3. Does the change affect any risk control?
4. Does the change alter supported platforms, browsers, hosting, or dependencies?
5. Does the change affect model behavior, prompts, or provider outputs?
6. Does the change affect cybersecurity, privacy, or audit traceability?
7. Does the change require IFU, labeling, CER, PMS, or risk file updates?

## Model and prompt changes

Model and prompt changes are never treated as trivial in the regulated build.

Minimum requirements:

- documented rationale
- output impact assessment
- regression testing against representative validation cases
- risk review
- approval before production release

## Release approval

No release may enter production until:

- required verification is complete
- required validation updates are complete
- documentation impact is handled
- risk review is signed off
- release record identifies the exact configuration deployed

## Production drift control

Production must be monitored to ensure it matches the validated baseline:

- no unapproved settings changes
- no unapproved provider switches
- no unapproved prompt edits
- no unapproved dependency drift where safety/performance is affected

## Records generated by this procedure

- change request record
- impact assessment
- risk review update
- verification / validation evidence
- release approval record
- production configuration record
